# test_project_cl_2026
Это тестовый проект

## Триаж машинного перевода для постредактирования (QE), zh → ru

Задача: по паре (китайский исходник, машинный перевод на русский), **без
эталонного перевода**, предсказать, насколько плох результат MT и нужно ли
его показывать переводчику для постредактирования — или можно принять
почти как есть.

### Структура проекта

| Файл | Назначение |
|---|---|
| `02_mt_triage_qe_colab.ipynb` | Colab-ноутбук: сбор zh-ru пар (UN Parallel Corpus), генерация MT (`facebook/nllb-200-distilled-600M`), авторазметка качества (chrF + LaBSE similarity, только для сборки обучающей выборки), извлечение признаков без эталона (jieba, длины, NER-несоответствия, cross-lingual LaBSE src↔mt), быстрая статистика на Python и ML-модели (RandomForest/GradientBoosting: регрессия качества + классификация accept/edit). На выходе — `features.csv`. |
| `03_stat_analysis.R` | Статистический блок на R поверх `features.csv`: корреляции Пирсона признаков с `quality_score`, t-test между группами accept/needs_edit, линейная регрессия (`lm`), логистическая регрессия (`glm`), ANOVA, диагностические графики в `plots/`. |
| `data/sample_features.csv` | Синтетический пример с той же структурой, что и реальный `features.csv` — чтобы `03_stat_analysis.R` можно было проверить сразу, не запуская ноутбук. |
| `requirements.txt` | Python-зависимости пайплайна (для локального запуска; в самом ноутбуке есть свой `!pip install`). |
| `01_first_script.R` | Исходный тестовый файл проекта (не относится к QE-пайплайну). |

### Как запустить

1. **Ноутбук** — откройте `02_mt_triage_qe_colab.ipynb` в Google Colab (File →
   Upload notebook, либо File → Open notebook → GitHub) и выполните ячейки
   по порядку. На выходе получите `features.csv`.
2. **R-анализ** — скопируйте `features.csv` в корень проекта и запустите:

   ```r
   Rscript 03_stat_analysis.R
   ```

   Если `features.csv` рядом нет, скрипт автоматически возьмёт
   `data/sample_features.csv` — так его можно проверить сразу после клонирования
   репозитория, без Colab.

### Как читать признаки

Все признаки в `features.csv` считаются **только** из пары (исходник, MT) —
без обращения к эталону, поэтому они валидны и на инференсе для нового,
ещё не отредактированного контента:

- `zh_len_chars`, `zh_len_tokens`, `zh_rare_ratio` — длина и «редкость»
  китайского исходника (jieba-токенизация, доля токенов, встречающихся в
  корпусе ≤ 2 раз).
- `mt_len_chars`, `mt_len_tokens`, `len_ratio` — длина MT-перевода и её
  отношение к длине исходника.
- `digit_mismatch` — расхождение количества чисел между исходником и MT.
- `zh_n_entities`, `mt_n_entities`, `entity_mismatch` — число именованных
  сущностей (spaCy) в исходнике/переводе и их расхождение.
- `labse_sim_src_mt` — cross-lingual косинусная близость эмбеддингов
  (LaBSE) исходника и MT-перевода.
- `quality_score`, `needs_edit` — целевые переменные, полученные на этапе
  авторазметки (шаг 3 ноутбука) через сравнение MT с человеческим
  эталоном; в проде это то, что модель учится предсказывать.
